Meet the Server Crew

Name the main actors so you know where to put new features, fixes, and protocol decisions.

Think of this codebase like a film set

On a film set, the director, camera crew, editor, and sound team all touch the same scene, but each role stays narrow. This server does the same thing with architecture: one part handles startup, another handles networking, and another handles request-time decisions.

This is the vocabulary module. When you tell AI "put that in middleware" or "that belongs in the protocol handler," you are really making an architecture decision.

🎬
Aha: small roles make AI prompts sharper

When components have clear jobs, your prompts get precise: "add caching middleware" is much better than "change the server stuff somewhere."

These are the features the builder plugs in

When the builder wires the app, it is deciding which reusable behaviors every request can use. That is a different kind of decision from request-time business logic.

🪪

Auth

Short for authentication and authorization work. Use this layer when AI should identify the user or check whether a request is allowed.

🧾

Session

A session keeps a little state around between requests.

📁

Static Files

Static files are ready-made files the server can hand back directly.

🧭

MVC

MVC is the structured page-and-handler layer for larger app behavior.

1️⃣

HTTP/1.1

One supported HTTP version. The factory can choose a handler for it.

2️⃣

HTTP/2

Another supported rulebook. Keeping version choice isolated makes the rest of the app simpler.

Why this matters for AI prompts

If you want a reusable caching rule, ask for middleware. If you want a new page pattern, ask about MVC. If you want a new protocol, ask about the factory and handler layer.

See startup wiring and protocol choice in real code

These two snippets show the split clearly: first, startup adds reusable capabilities to the app; later, the factory chooses the right protocol specialist for each connection.

CODE 
        appBuilder.UseAuthorization(options =>
        {
            options.Policies.Add("is-mini-web-server", policy => policy.RequireClaimValue(ClaimTypes.NameIdentifier, "mini-web-server"));
            options.Policies.Add("dev-required", policy => policy.RequireClaimValue(ClaimTypes.Role, "Developer"));

            options.Routes.Add("/helpcheck", "is-mini-web-server", "dev-required");
        });

        appBuilder.UseSession();
        appBuilder.UseStaticFiles("wwwroot", defaultMaxAge: 7 * 24 * 3600); // defaultMaxAge = 7 days
        appBuilder.UseMvc();
PLAIN ENGLISH

Start the authorization setup and hand it a configuration object.

Create reusable policies that check for specific claims.

Attach those rules to the /helpcheck route so the route gets protection automatically.

Then add sessions, static-file serving, and MVC as more reusable app capabilities.

This is classic startup wiring: decide what powers the app has before live traffic starts.

CODE 
        if (httpVersion == HttpVersions.Http11)
        {
            // in reality we often use default parsers

            return new Http11ProtocolHandler(config, loggerFactory,
                services.GetService<IHttpComponentParser>() ?? new ByteSequenceHttpParser(loggerFactory),
                services.GetService<ICookieValueParser>() ?? new DefaultCookieParser(),
                protocolHandlerContext
                );
        }
        else if (httpVersion == HttpVersions.Http20)
        {
            return new Http2ProtocolHandler(loggerFactory,
                services.GetService<IHttpComponentParser>() ?? new ByteSequenceHttpParser(loggerFactory),
                services.GetService<ICookieValueParser>() ?? new DefaultCookieParser(),
                protocolHandlerContext
                );
        }
PLAIN ENGLISH

If the connection is HTTP/1.1, build an HTTP/1.1 protocol handler.

Try to reuse shared services for the parser pieces; if none are registered, fall back to default ones.

Pass in the handler context so the new object can read from the live connection.

If the connection is HTTP/2 instead, return the HTTP/2 specialist instead of forcing one class to handle every version.

That separation is what lets the server grow new protocol support without bloating one giant class.
