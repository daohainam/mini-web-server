Why This Server Is Built This Way

The design choices that make this server flexible, cheaper to run, and much easier to steer with AI.

A warehouse with specialist teams

Imagine a shipping warehouse that does protocol checks, package reading, and delivery by handing each job to a different team instead of forcing one exhausted worker to do everything.

That is the core idea here: split the server into focused parts so you can swap, test, and debug one part without tearing apart the whole machine.

📦

All-in-one server

Reads everything immediately, mixes parsing with routing, and makes every change feel risky.

one giant flow Fast to start, hard to evolve
tight coupling One change can ripple everywhere
🏗️

This modular server

Separates middleware, parsers, handlers, and body reading so each piece can stay simple.

focused roles Each layer has one main job
easier swaps You can replace one piece without rewriting the rest
💡
Why this matters for AI steering

If you can say “keep the parser separate from the request handler” or “make body reading lazy,” you give AI a much sharper instruction than “clean this up.”

Four patterns hiding in plain sight

This module is really a vocabulary upgrade. These are the pattern names you can now use when you want AI to copy the same architecture on purpose.

🐢

Lazy Body Read

Do not pull the whole request body up front. Wait until the app actually needs it.

🛣️

Protocol-Specific Handler

Give HTTP/1.1 and HTTP/2 their own paths so each version can behave correctly.

🧱

Builder Pattern

Use a builder object to assemble a request piece by piece instead of in one giant constructor call.

🪜

Abstraction Layer

An abstraction layer keeps the rest of the app from caring which concrete parser or cookie reader is underneath.

Why the server waits before pulling the big box

The request body might be tiny, or it might be huge. This server starts reading it in parallel, lets the app do its work, then stops wasting effort if the work is already done.

Now the server moves from headers into body-reading work.

It creates a linked cancellation token so body reading can be stopped cleanly.

One task starts reading the body while another task runs the app logic.

Then the stop signal is triggered, because the app may finish before all body reading is worth doing.

The comment openly admits there is still design work here, which is useful debugging honesty.

Finally, the server waits for both jobs so the connection does not race ahead half-finished.

PLAIN ENGLISH

This method is the dedicated body reader.

The comments state the design rule clearly: do not read a huge body until the app really needs it.

That choice keeps the server responsive and avoids spending resources on bad requests.

It only plans to read the exact number of bytes promised by Content-Length.

If there is no body at all, it immediately marks the body pipeline complete and moves on.

Different protocol roads, same destination

A handler factory keeps the high-level server calm while still letting HTTP/1.1 and HTTP/2 behave differently underneath.

CODE 
        if (httpVersion == HttpVersions.Http11)
        {
            // in reality we often use default parsers

            return new Http11ProtocolHandler(config, loggerFactory,
                services.GetService<IHttpComponentParser>() ?? new ByteSequenceHttpParser(loggerFactory),
                services.GetService<ICookieValueParser>() ?? new DefaultCookieParser(),
                protocolHandlerContext
                );
        }
        else if (httpVersion == HttpVersions.Http20)
        {
            return new Http2ProtocolHandler(loggerFactory,
                services.GetService<IHttpComponentParser>() ?? new ByteSequenceHttpParser(loggerFactory),
                services.GetService<ICookieValueParser>() ?? new DefaultCookieParser(),
                protocolHandlerContext
                );
PLAIN ENGLISH

First, check which HTTP version this connection is using.

If it is HTTP/1.1, build an Http11ProtocolHandler with the shared dependencies it needs.

If the app did not provide a custom parser or cookie parser, fall back to the default ones.

If it is HTTP/2, create the HTTP/2-specific handler instead.

This is the payoff of modular design: version-specific behavior changes without forcing the whole server to fork.
