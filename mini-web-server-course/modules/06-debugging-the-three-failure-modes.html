Debugging the Three Failure Modes

A calm way to classify failures: unreadable input, missing destination, or a crash during execution.

Three warning lights, not endless chaos

Server bugs feel random until you notice this codebase keeps most failures inside three lanes: 400 for a broken read, 404 for a missing route, and 500 for a crash while work is already underway.

That map matters because it tells you what to ask AI next: “check the parser,” “check route selection,” or “check the handler exception path.”

400

Read lane

The server could not successfully parse the incoming request, so it sends Bad Request and closes the bad connection.

Ask AI Inspect parser input, headers, and request formatting first
404

Find lane

The request was readable, but no matching handler claimed it, so the response becomes Not Found.

Ask AI Trace endpoint registration and route matching
500

Execute lane

The right handler started running, but an exception happened inside that work.

Ask AI Look at the action code, logs, and caught exceptions

Lane 1: unreadable input becomes 400

If the server cannot read the request cleanly, it does not keep pretending. It sends a 400 and shuts down that broken conversation.

CODE 
                    var readRequestResult = await protocolHandler.ReadRequestAsync(requestBuilder, cancellationToken);
                    if (!readRequestResult)
                    {

                        var response = new HttpResponse(HttpResponseCodes.BadRequest, config.ClientStream);

PLAIN ENGLISH

Try to read the incoming request using the protocol handler.

If that read fails, stop pretending this connection is healthy.

Turn off keep-alive so the bad connection does not hang around.

Build a 400 Bad Request response on the client stream.

Set a send timeout, log what is happening, and send the response back quickly.

CODE 
    private static async Task<ReadRequestTestResult> ReadRequestAsync(string requestContent)
    {
        var reader = PipeUtils.String2Reader(requestContent);
        var loggerFactory = new LoggerFactory();

        var httpParser = new ByteSequenceHttpParser(loggerFactory);
        var handler = new Http11ProtocolHandler(new ProtocolHandlerConfiguration(global::MiniWebServer.Abstractions.HttpVersions.Http11, 1024 * 1024 * 10), loggerFactory, httpParser, new DefaultCookieParser(), 
            new ProtocolHandlerContext() { 
                PipeReader = reader, Stream = new MemoryStream()
            });

        var requestBuilder = new HttpWebRequestBuilder();

        var result = await handler.ReadRequestAsync(requestBuilder, CancellationToken.None);
        return new ReadRequestTestResult(result, requestBuilder);
PLAIN ENGLISH

This test creates a reader from a raw request string so the parser can be exercised in isolation.

It builds the HTTP parser and a specific HTTP/1.1 protocol handler with a fake stream.

Then it creates a fresh request builder to collect parsed fields.

The real check is one call: can the handler read this request successfully or not?

That is powerful debugging design: isolate one subsystem, feed it controlled input, and inspect the result without booting the whole server.

🧪
Why isolated tests help

When AI gets stuck in a bug loop, asking it to reproduce the problem in a tiny parser-only test is often the fastest escape hatch.

Lane 2 and Lane 3: not found versus crashed

Once the request is readable, the next question is simple: did nobody claim the job, or did the worker crash while doing it?

CODE 
                try
                {
                    await action.InvokeAsync(context, cancellationToken);
                }
                catch (Exception ex)
                {
                    logger.LogError(ex, "[{cid}] - Error executing action handler", ConnectionId);
                    response.StatusCode = HttpResponseCodes.InternalServerError;
                }
            }
            else
            {
                StandardResponseBuilderHelpers.NotFound(response);
PLAIN ENGLISH

If an action exists, the server tries to run it.

If running that action throws an exception, the error is logged and the response becomes 500 Internal Server Error.

If there was no action to run in the first place, the code falls into the else branch instead.

That else branch is the key distinction: unreadable input is no longer the problem here; missing ownership is.

CODE 
public class StandardResponseBuilderHelpers
{
    public static void NotFound(IHttpResponse response)
    {
        response.StatusCode = HttpResponseCodes.NotFound;
        response.Headers.ContentLength = 0;
    }
PLAIN ENGLISH

This helper gives the response a 404 Not Found status.

It also sets the content length to zero, which means there is no body payload attached.

That is a clean “nobody claimed this request” answer, which is much more precise than blaming the server with a 500.
