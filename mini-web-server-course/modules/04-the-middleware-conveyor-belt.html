The Middleware Conveyor Belt

Inside a Mini Web Server: once a request is parsed, the next question is who gets first crack at handling it.

Order is the architecture here

A middleware chain is like an airport conveyor belt: every station gets a chance to act, and the order changes the outcome.

Authentication should usually happen before authorization. Session, static files, and MVC each compete for the same request.

STARTUP ORDER 
        appBuilder.UseAuthorization(options =>
        {
            options.Policies.Add("is-mini-web-server", policy => policy.RequireClaimValue(ClaimTypes.NameIdentifier, "mini-web-server"));
            options.Policies.Add("dev-required", policy => policy.RequireClaimValue(ClaimTypes.Role, "Developer"));

            options.Routes.Add("/helpcheck", "is-mini-web-server", "dev-required");
        });

        appBuilder.UseSession();
        appBuilder.UseStaticFiles("wwwroot", defaultMaxAge: 7 * 24 * 3600); // defaultMaxAge = 7 days
        appBuilder.UseMvc();
/helpcheck Protected route that needs both named policies.
is-mini-web-server Requires a matching name identifier claim.
dev-required Requires the Developer role before access is allowed.
wwwroot Static file root with a seven-day default cache age.

This builder makes the last added middleware run first

⚠️
Core instinct to keep

This server uses LIFO wrapping. Changing startup order changes runtime behavior.

CODE 
        if (typeof(IMiddleware).IsAssignableFrom(middlewareType))
        {
            middlewareTypes.Insert(0, middlewareType); // LIFO: last added middleware will be called first
        }
PLAIN ENGLISH

Only real middleware types enter this special chain.

Instead of appending to the end, the builder inserts each new item at the front of the list.

That means the middleware you add last in startup code becomes the one that gets called first when a request arrives.

CODE 
        foreach (var middleware in middlewareChain)
        {
            var callWrapper = new MiddlewareWrapper(middleware, callable);

            callable = callWrapper;
        }

        return callable;
PLAIN ENGLISH

Walk through the chain one middleware at a time.

Each pass creates a wrapper that knows two things: who should run now, and who should be called next.

Then that wrapper becomes the new callable, so the next loop wraps around it again.

By the end, the outermost wrapper is the first station the request hits.

/logo.svg Should get served early by Static Files.
404 Only the fallback should answer when every other station passed.
