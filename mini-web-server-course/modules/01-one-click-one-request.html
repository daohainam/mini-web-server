One Click, One Request

Follow one request journey from a browser address bar to the server answer coming back.

A tiny request journey you can actually trace

Imagine you open a browser and visit /string-api/toupper?text=hello. That sends a request to a server route with a query parameter.

This matters for AI steering because every "add a new endpoint" or "trace the bug" prompt depends on knowing which station in the trip owns the next move.

📦
Aha: servers work by handoffs

A server is rarely one giant magic function. It is a chain of small parts that each touch the parcel once.

The six-hop route map

1
The browser sends the path and the text value

The request journey starts with a URL that asks for one very specific job: turn text into uppercase.

2
The listener notices a new connection

A listener accepts a new TCP client and gives it an ID.

3
The connection object checks the incoming format

This is where the code decides which HTTP version it is dealing with.

4
A protocol handler reads the raw bytes

A protocol handler interprets the bytes instead of guessing.

5
The route handler builds the answer

The app code reads the text input, transforms it, and prepares a response.

6
The answer travels back the same line

If you ever get the wrong result, you can debug by finding the last hop that still looked healthy.

Open the two most important code moments

The route snippet is where the visible answer is created. The connection snippet is where the server decides how to interpret the wire-level message before any route logic can run.

CODE 
        app.MapGet("/string-api/toupper", (context, cancellationToken) =>
        {
            string p = context.Request.QueryParameters["text"].Value ?? string.Empty;

            context.Response.Content = new MiniApp.Content.StringContent(p + " ===> " + p.ToUpper());

            return Task.CompletedTask;
        });
PLAIN ENGLISH

Create a GET route for one exact path: /string-api/toupper.

When that path is hit, open a handler with the request context already attached.

Pull out the text input. If it is missing, quietly use an empty string instead of crashing.

Build the answer content by joining the original text and its uppercase version.

Finish the handler. There is no extra background work to wait for here.

PLAIN ENGLISH

If the connection is speaking HTTP/2, skip past its special opening marker before reading the rest.

Build a small config object that says which HTTP version we saw and how large a request body is allowed.

Create a handler context that carries the stream and the reader that pulls bytes from it.

Ask the factory to create the right protocol handler for this connection, instead of hard-coding one reader for every case.
